# 意图对齐

用户描述问题时，常常只描述结果，不描述触发链路。先把问题对齐到“可复现动作 + 预期 + 实际 + 证据”。

## 什么时候读

- 用户只说“登录失败”“页面不对”“为什么不行”。
- 问题可能有多种含义，直接动手会容易修偏。
- 用户给了截图或报错，但缺少操作步骤。

## 最小问题模板

优先问 1-3 个短问题：

```text
1. 你是在哪个页面/入口触发的？
2. 你做了哪一步后出问题？
3. 屏幕上看到的是报错、空白、卡住，还是数据不对？
```

如果已有足够线索，不要为了模板而追问，直接复现。

## ASCII 对齐模板

```text
[用户现象]
用户点击/输入/打开 <入口>
  -> <系统应该做什么>
  -> 实际变成 <空白 / 报错 / 卡住 / 数据错误>

[需要确认]
A. 事件没有触发
B. 请求没有发出
C. 请求发出但失败
D. 请求成功但状态没有更新
E. 页面运行时崩溃
```

## 对齐记录

```text
问题描述：<用户原话>
入口：<页面 / API / 命令 / 扩展页面>
触发条件：<操作步骤>
预期行为：<应该发生什么>
实际行为：<实际发生什么>
关键证据：<截图 / 报错 / console / network / 日志>
初步分类：<UI / API / 状态 / 构建 / 环境 / 外部服务>
```

## 新手友好解释

如果用户是零基础，把报错翻译成人话：

```text
报错原文：TypeError: Cannot read properties of undefined
人话翻译：代码想从一个“空盒子”里拿东西，但盒子还没有准备好。
定位方式：先看是哪一行拿了这个值，再往前查这个值应该从哪里来。
```
