# 动态补洞策略

不要按固定顺序机械完成澄清题。先吸收用户材料，再围绕六个 PRD 必需维度评估充分度；只追问缺失、冲突或不够具体且会影响 MVP 边界的维度。最多问六道正式澄清题，信息足够时立即停止追问。

六个维度：

1. 工具形态
2. 目标用户
3. 使用场景
4. 核心痛点
5. MVP 功能
6. 交互流程

## 信息充分度总规则

每个维度只允许标记为：

- `已充分`：能直接写进 PRD，并能指导 MVP 边界、页面结构或交互流程。
- `不够具体`：材料提到了方向，但粒度太粗，仍无法排除相近路线。
- `缺失`：材料没有覆盖该维度。
- `冲突`：材料里存在互相矛盾的说法，或用户要求与后续 CodeNow 工具形态映射冲突。

关键判断：

- 提到关键词不等于充分。`小红书博主` 是用户大类，不等于第一版目标用户充分；`网站` 是入口大类，不等于工具形态充分；`从评论里提炼选题` 是功能方向，不等于使用场景充分。
- 只有当一个维度具体到「能排除相近替代方向」时，才算 `已充分`。
- 如果材料中已有的信息只能让你猜一个方向，但猜错会改变核心页面、功能优先级或技术形态，必须追问。
- 追问不是为了收集更多细节，而是为了降低会影响 MVP 边界的关键不确定性。

## 追问优先级

默认优先级：

1. 工具形态
2. 目标用户
3. 使用场景
4. 核心痛点
5. MVP 功能
6. 交互流程

这不是固定顺序。如果更高优先级已经 `已充分`，就跳过它；如果低优先级存在关键冲突，也可以提前追问。

每次只追问当前最影响 MVP 边界的一个维度。用户回答后，立即重新评估全部维度，不要沿用旧判断。

## 材料吸收规则

### 文本和 Markdown

- 提取明确事实：产品类型、服务对象、平台、功能、使用场景、竞品、限制条件。
- 提取隐含信号：用户反复强调的痛点、想避免的做法、已有业务流程。
- 标记不充分信号：泛称、口号、功能名堆叠、没有触发时刻的场景描述。

### 图片

先判断图片更像哪种材料，再提取信号：

- 产品截图：提取页面结构、入口、信息层级、已有功能。
- 线框图 / 手写草图：提取模块关系、操作路径、核心对象。
- 竞品截图：提取可借鉴能力，但不要默认照抄为 MVP 主线。
- 视觉参考图：提取风格信号；若基础版 PRD 未生成，不要为了设计系统延迟 PRD。

如果图片含义不确定，追问图片用途，例如「这张图更像功能参考，还是视觉风格参考？」。

## 1. 工具形态

目的：确认产品的主要交付入口、首选设备和平台能力边界。工具形态会影响后续用户、场景、功能、交互流程和 ASCII 流程图。

### 充分标准

只有能映射到以下四类之一，才算 `已充分`：

- `网站，优先在桌面使用`
- `网站，优先在移动端使用`
- `桌面软件`
- `浏览器插件`

可视为充分的表达：

- `桌面端网页`、`PC 网站`、`后台管理网页` -> `网站，优先在桌面使用`
- `手机网页`、`H5`、`移动端网页`、`移动优先 Web` -> `网站，优先在移动端使用`
- `Mac 桌面软件`、`Windows 客户端`、`Electron 应用` -> `桌面软件`
- `Chrome 插件`、`浏览器扩展`、`划词插件` -> `浏览器插件`

必须追问的表达：

- `网站`：不知道桌面网页还是移动网页。
- `App`：不知道用户是泛指产品、手机 App、桌面 App，还是移动网页。
- `平台`、`工具`、`系统`：没有交付入口和设备边界。
- `小程序`、`原生 iOS/Android App`：与当前后续开发映射不完全一致，需要确认是否接受四类 MVP 形态中的替代方案。

推荐问题：

```text
你希望第一版优先以哪种形态交付？
```

固定选项：

- A. 网站，优先在桌面使用
- B. 网站，优先在移动端使用
- C. 桌面软件
- D. 浏览器插件

模式：单选。

## 2. 目标用户

目的：锁定第一版最优先服务的人群，避免 PRD 同时服务多个相近但需求不同的人。

### 充分标准

目标用户达到以下条件才算 `已充分`：

- 能说清第一版主要服务哪一类具体人群。
- 能排除相近但不同的用户群。
- 用户身份与工具形态、使用场景、痛点能连起来。

不充分示例：

- `小红书博主`：需要继续确认是新手博主、垂类博主、品牌运营、内容团队、接商单博主等。
- `学生`：需要确认学段、任务、使用压力和使用入口。
- `职场人`：需要确认岗位、工作流和高频任务。
- `创作者`：需要确认内容类型、规模、收入模式或生产阶段。

推荐问题结构：

```text
第一版更应该优先服务哪一类 {{用户大类}}？
```

选项模式：

- 具体职业 / 身份人群
- 人生阶段人群
- 以行为定义的人群
- 高紧迫度的小众人群

模式：单选。

## 3. 使用场景

目的：确认用户在什么时刻、因为什么触发、打开产品完成什么动作。使用场景直接决定场景故事、主界面入口和交互流程。

### 充分标准

使用场景达到以下条件才算 `已充分`：

- 有具体触发时刻或事件。
- 有用户打开产品的原因。
- 有用户在产品内完成的核心动作。
- 场景符合工具形态的入口和设备边界。

不充分示例：

- `从评论里提炼爆款选题`：这是功能方向，不是场景；需要知道用户是在发文前找选题、爆文后复盘评论、做周更排期，还是接品牌合作时找角度。
- `学习英语`：需要知道是在浏览网页、读论文、刷题、通勤背词，还是写作修改。
- `管理客户`：需要知道是在录入线索、跟进商机、复盘成交，还是提醒回访。

推荐问题结构：

```text
用户最常在什么时候打开这个产品？
```

选项模式：

- 具体时间与地点
- 某个触发事件
- 重复发生的工作流节点
- 紧急或情绪波动时刻

模式：多选。

## 4. 核心痛点

目的：明确产品第一版必须解决的问题。痛点要能解释用户为什么需要这个产品，而不是只描述功能。

### 充分标准

核心痛点达到以下条件才算 `已充分`：

- 痛点发生在已确认目标用户和使用场景中。
- 痛点具体到可被功能直接缓解。
- 痛点能区分优先级，不是一串泛泛愿望。

不充分示例：

- `提高效率`：需要知道慢在哪里、为什么慢、慢会造成什么损失。
- `智能推荐`：这是方案，不是痛点。
- `做爆款`：需要知道是选题不准、评论洞察难、标题包装弱，还是内容排期乱。

推荐问题结构：

```text
这些用户在这个场景里最需要先解决的问题是什么？
```

选项模式：

- 时间与效率类痛点
- 焦虑、不确定或决策类痛点
- 成本、获取渠道或资源类痛点
- 记录、记忆或整理类痛点

模式：多选。

## 5. MVP 功能

目的：圈定第一版要上的功能。功能必须能落地到界面和操作，不要停留在能力口号。

### 充分标准

MVP 功能达到以下条件才算 `已充分`：

- 至少能形成 3-4 个具体功能模块，或用户明确要求极简版本。
- 每个功能都能回应已确认痛点和场景。
- 每个功能都有清楚输入、处理或输出。
- 不把未确认的高级能力写成第一版核心。

不充分示例：

- `AI 助手`：不知道输入什么、输出什么、在哪个页面使用。
- `数据分析`：不知道分析对象、指标和结果用途。
- `个性化推荐`：不知道基于什么信号推荐、推荐给谁、推荐后做什么。

推荐问题结构：

```text
第一版 MVP 最需要包含哪些核心功能？
```

选项模式：

- 输入或采集类功能
- 推荐或生成类功能
- 引导动作类功能
- 历史、追踪或反馈类功能

模式：多选。

## 6. 交互流程

目的：确定从进入产品到获得价值的主路径。交互流程要能直接映射到 `【交互流程】` 和 ASCII 流程图。

### 充分标准

交互流程达到以下条件才算 `已充分`：

- 有明确入口。
- 有关键输入或选择动作。
- 有系统处理或生成结果。
- 有用户下一步动作或反馈闭环。
- 流程符合已确认工具形态。

不充分示例：

- `打开后使用 AI 生成`：缺入口、输入、结果和下一步。
- `上传数据后分析`：缺上传对象、分析结果和用户如何使用结果。
- `看推荐`：缺推荐触发、结果呈现和用户决策动作。

推荐问题结构：

```text
用户完成一次核心任务时，最自然的操作路径是什么？
```

选项模式：

- 推荐优先路径
- 搜索 / 筛选优先路径
- 引导式问卷路径
- 记录后再回顾路径

模式：默认单选；仅当产品天然存在多条入口路径时，可改为多选。

## 工具形态分支约束

确认工具形态后，后续追问和 PRD 必须遵守对应平台边界。

### 网站，优先在桌面使用

- 默认围绕大屏信息组织、仪表盘、表格/列表、批量操作、深度工作流和账号体系出题。
- 避免把移动碎片时间、系统级本地文件操作、浏览器插件入口作为默认主线。
- 目标用户选项应体现桌面办公、学习、分析、创作或管理场景。

### 网站，优先在移动端使用

- 默认围绕手机打开、轻量任务、即时反馈、触控交互、碎片时间和分享传播出题。
- 可以出现通勤、排队、睡前、路上等移动场景。
- 避免把复杂后台、长表格、系统级桌面能力或浏览器插件入口作为默认主线。

### 桌面软件

- 默认围绕本地文件、长时间工作、离线或半离线、系统资源、专业工作台和高性能处理出题。
- 目标用户选项应体现创作者、分析者、开发者、专业岗位或需要本地素材处理的人群。
- 避免把纯网页内容消费、移动碎片时间或浏览器插件入口作为默认主线。

### 浏览器插件

- 后续追问必须围绕浏览器内使用、网页上下文、插件弹窗、侧边栏、划词/选区、右键菜单、工具栏图标、Tab、网页内容读取、跨站点轻量存储等能力。
- 目标用户选项应以「经常在浏览器里完成某类任务的人」来定义，例如阅读英文网页、查资料、看论文、处理网页文本、跨站收藏整理的人。
- 痛点选项必须落在网页阅读/网页操作过程里，例如阅读流被查词打断、缺少当前网页语境解释、跨网站资料分散、划过的词句没有沉淀。
- 使用场景选项必须能发生在浏览器中，例如打开英文网页、选中段落、读在线 PDF、在多个标签页整理资料、从插件侧边栏回顾。
- MVP 功能选项优先考虑插件入口和浏览器能力，例如划词气泡、侧边栏解释、网页上下文摘要、生词自动收藏、当前页面知识卡片、跨站历史。
- 交互流程选项应以插件真实入口组织，例如「选中文本后弹出解释」「点击工具栏打开侧边栏」「右键菜单处理选区」「自动识别当前页面并给出建议」。
- 除非用户明确补充，不要把通勤者、移动端碎片学习、独立 App 打卡、桌面软件式文件管理作为默认选项。

## 动态补洞示例

用户输入：

```text
我想做一个给小红书博主用的选题网站，桌面端，帮她从评论里提炼爆款选题。
```

内部判断：

- 工具形态：`已充分`。`网站，桌面端` 可映射为 `网站，优先在桌面使用`。
- 目标用户：`不够具体`。`小红书博主` 是大类，无法决定第一版服务新手、垂类博主、品牌运营还是内容团队。
- 使用场景：`不够具体`。`从评论里提炼爆款选题` 是功能方向，不知道触发时刻。
- 核心痛点：`不够具体`。能推断选题困难，但不知道是找不到用户需求、爆文复盘难、品牌转化难还是排期混乱。
- MVP 功能：`不够具体`。只有一个功能方向，无法决定第一版模块组合。
- 交互流程：`缺失`。

第一道追问应优先补目标用户，例如：

```text
第一版更应该优先服务哪类小红书创作者？

A. 刚起号、缺稳定选题的新手博主
B. 有一定粉丝、想提高爆款率的垂类博主
C. 帮品牌账号做内容的运营人员
D. 同时管理多个账号的内容团队
```

随后根据用户答案，再判断是否追问使用场景，例如：

```text
用户最常在什么时候打开这个工具？

A. 发文前没有选题，想从评论里找下一篇内容
B. 一篇笔记爆了以后，想复盘评论里的真实需求
C. 每周做内容排期时，想批量整理可写选题
D. 接到品牌合作后，想从评论里找到更容易转化的角度
```

## 选项撰写规则

- 每题始终恰好四个选项。
- 选项要短到能一眼扫完，又要具体到足以塑造 PRD。
- 避免空洞选项，如「提高效率」「优化体验」「智能化」。
- 优先带用户情境的选项：例如「发文前 30 分钟从评论找下一篇角度」优于「选题场景」。
- 不要重复已充分的信息，除非选项是在进一步收窄范围。
- 多选题要提醒用户：可以选不止一个字母。
- 每题选项生成后自检：是否命中「初始想法」「已确认工具形态」「当前待补洞维度」三类信号；如果没有，重写选项。
- 自检是否误把未选项当作主线；如果是，重写题干或替换该选项。
- 如果用户自定义答案比选项更清楚，以用户自定义答案为准，并重新评估全部维度。
