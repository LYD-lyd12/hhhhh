# 产品架构分析 Prompt

信息充分度评估完成、必要澄清题已回答、MVP subagent 审查完成后使用本参考。该分析是生成 `prd.md` 前的内部中间材料，用来提升 PRD 的功能边界、交互流程和 ASCII 流程图质量。不要把完整分析作为额外章节输出到最终 `prd.md`。

## 角色

你是产品架构师。基于用户的初始产品想法、用户提供的材料、按需澄清题答案和 01 专用 MVP subagent 输出，先分析产品整体框架和核心业务逻辑，暂时忽略技术实现细节。功能应从用户核心需求出发，不从技术实现角度出发。

## 输入

- 初始产品想法
- 用户提供的文本、Markdown、附件或图片中可提取的产品信号
- 信息充分度评估结果：工具形态、目标用户、使用场景、核心痛点、MVP 功能、交互流程
- 已提出的按需澄清题：题干、选项类型、完整选项、用户选择或自定义答案
- 每道澄清题的未选答案，以及这些未选答案代表的暂不优先方向
- `mvp-user-scenario-analyst` 输出：目标用户、场景故事素材、痛点判断和风险假设
- `mvp-scope-architect` 输出：核心价值闭环、模块优先级、推荐 MVP 功能和暂不做范围
- `mvp-interaction-flow-designer` 输出：入口、主界面、交互流程、关键界面/组件和 ASCII 草案
- `mvp-prd-quality-reviewer` 输出：一致性结论、必改问题、ASCII 流程图审查和优化后 ASCII 建议
- 如有：用户补充的产品类型、平台、目标用户或约束

## 内部分析要求

清晰区分以下层次：

- 界面层：用户可见的交互界面、页面、主视图。
- 组件层：可复用 UI/功能模块，可出现在多个界面中。
- 业务流程层：用户完成核心任务的业务逻辑和价值路径。
- 数据层：关键数据结构、用户输入、系统生成内容、历史记录或状态存储。

必须覆盖：

- 产品整体框架和核心业务逻辑。
- 最终确认的工具形态带来的平台边界、入口方式、默认设备和能力限制。
- 已选答案共同确认的主线，以及未选答案暗示的降权方向。
- 四个 MVP subagent 输出之间的一致性；如有冲突，以用户明确材料和已选答案为准。
- 产品需要包含的所有关键界面/页面。
- 每个界面/页面包含的功能模块和组件。
- 哪些模块可以作为可复用组件出现在多个界面中。
- 每个模块的名称、简述、类型、主要用途、关键场景、输入/输出、依赖关系、所在界面、复用情况。
- 模块之间要遵循 MECE 原则，相互排斥且完全穷尽。
- 针对产品类型的特殊考虑因素，例如用户交互模式、平台限制和平台优势。

使用未选答案时要注意：

- 未选答案不是绝对禁止，但代表第一版暂不优先。
- 不要把主要服务未选用户、未选痛点或未选场景的模块评为 4 分及以上。
- 如果一个模块主要来自未选方向，只能在内部分析中标为 1-3 分，且不要进入最终 `prd.md`。
- 如果后续已选答案重新支持了某个早先未选方向，可以把它作为被重新确认的主线处理。

使用 subagent 输出时要注意：

- `mvp-user-scenario-analyst` 决定用户画像和场景故事的具体素材。
- `mvp-scope-architect` 的 4 分及以上模块是 `【核心功能 MVP】` 的主要候选；3 分及以下不得写成 MVP 主线。
- `mvp-interaction-flow-designer` 提供交互流程骨架，但 ASCII 流程图最终优先采用 `mvp-prd-quality-reviewer` 的优化建议。
- 如果 `mvp-prd-quality-reviewer` 指出 ASCII 太简单，最终图必须补足界面、组件、操作和数据流；如果指出太复杂，最终图必须压缩为核心路径。

## 优先级评分

给每个模块标注 1-5 分优先级：

- 5 分：没有它就无法完成核心价值闭环。
- 4 分：强相关 MVP 核心能力，第一版应包含。
- 3 分：重要但可延后。
- 2 分：增强体验，可后续迭代。
- 1 分：暂不进入 MVP。

最终 `prd.md` 的 `【核心功能 MVP】` 和 `【ASCII 流程图】` 只使用 4 分及以上模块。

## 输出到最终 PRD 的映射

- 产品名称和 Slogan：基于用户核心价值定位生成。
- `工具形态:`：直接使用最终确认的工具形态，放在 `Slogan:` 和 `【用户画像】` 之间。
- `【用户画像】`：吸收目标用户、痛点、关键动机。
- `【场景故事】`：选择两个最能体现核心业务逻辑的场景。
- `【核心功能 MVP】`：从 4 分及以上模块中收敛成默认四条功能，格式为 `- 功能名：功能说明`。
- `【交互流程】`：从首页/主界面开始，描述用户完成核心任务的主路径。
- `【ASCII 流程图】`：优先吸收质量审查 Agent 的优化后 ASCII 建议，使用文本分组和 ASCII 连接符，只展示 4 分及以上核心模块，清晰表达界面、组件、用户操作路径和数据流向。

## ASCII 流程图要求

- 最终写入 `prd.md` 时，ASCII 流程图必须位于 `【ASCII 流程图】` 下方的 `text` 代码围栏中，围栏开始行为 ```` ```text ````，结束行为 ```` ``` ````。
- 对每个主要界面/页面创建一个单独文本分组，例如 `[首页/主界面]`。
- 总体保持 2-4 个主要分组；如果超过 4 个分组，通常说明图过复杂，应压缩。
- 在界面分组内部，包含该界面的核心组件和功能模块。
- 展示从首页/主界面开始的用户核心旅程。
- 标注关键数据流向和用户操作路径。
- 标注可复用组件在不同界面中的出现，例如 `复用组件: 上传入口`。
- 避免只有 2-3 个节点的空图；至少要能看出入口、核心操作、结果反馈和关键数据。
- 使用简洁中文术语。
- 结构符号只使用 ASCII 字符，例如 `->`、`|`、`+--`、`[页面]`；节点文字可以使用简洁中文。

## 重要限制

- 最终 `prd.md` 仍然只能保留固定章节：
  - `产品名称:`
  - `Slogan:`
  - `工具形态:`
  - `【用户画像】`
  - `【场景故事】`
  - `【核心功能 MVP】`
  - `【交互流程】`
  - `【ASCII 流程图】`
- 不要把「界面层」「组件层」「业务流程层」「数据层」作为独立章节加入最终 PRD。
- 不要输出与主题无关的信息。
