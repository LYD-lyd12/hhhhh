# MVP PRD 构建器

## 概述

通过「材料吸收 + 信息充分度评估 + 最多六道按需澄清题 + MVP subagent 审查」，把模糊的产品想法变成 `prd.md`。如果用户只唤起技能但没有给出产品想法，先询问用户想做什么，并给出灵感示例。信息足够后，先让 01 专用 MVP subagent 做用户场景、范围、交互和 ASCII 质量审查，再做内部产品架构分析，并把分析结果压缩进固定 PRD 模板。必须严格保留最终格式；不要新增章节、改标题，也不要在输出里解释文档本身。设计参考图分析是可选后置能力，不计入澄清题。

**已有 `prd/` 或活跃项目的日常想法增量**：用 **`product-discovery`**，不要用本 Skill 的冷启动 PRD 流程。

## 工作流

1. 如果用户没有提供初始产品想法，先执行「空启动流程」，不要直接问工具形态。
2. 收集用户已经给出的所有材料：对话文本、Markdown 文件、可读取附件和图片。
3. 如果用户附带图片，先判断图片类型：产品截图、线框图、竞品截图、视觉参考图或手写草图；图片可用于提取功能、界面、交互、场景或设计风格信号。
4. 阅读 `references/question-strategy.md`，按六个维度建立内部「信息充分度评估」：工具形态、目标用户、使用场景、核心痛点、MVP 功能、交互流程。
5. 每个维度只允许标记为以下状态之一：`已充分`、`不够具体`、`缺失`、`冲突`。
6. 只有 `已充分` 的维度可以跳过；材料里出现关键词不等于充分。例：`小红书博主` 仍需确认是哪类博主；`网站` 仍需确认桌面网页还是移动网页；`从评论里提炼选题` 不能自动等同核心使用场景。
7. 若存在 `不够具体`、`缺失` 或 `冲突`，优先追问当前最影响 MVP 边界的一个维度。默认优先级：工具形态 > 目标用户 > 使用场景 > 核心痛点 > MVP 功能 > 交互流程；若某个维度已经充分，则跳过它。
8. 每次只问一道选择题，最多六道。每道题提供恰好四个选项，标明单选或多选，必须允许用户填写自定义答案。
9. 每次用户回答后，重新评估六个维度；当六个维度都达到 `已充分`，立即停止追问，不要为了凑题数继续提问。
10. 若已问满六道仍存在普通缺口，使用材料和问答里的强信号做保守推断；若仍存在会导致 PRD 方向互相矛盾的关键冲突，不生成猜测版 PRD，先请用户用一句话澄清冲突点。
11. 信息足够后，生成内部 MVP Context Pack，包含用户材料摘要、六个维度充分度、最终确认信息、澄清题答案和未选答案降权信号。
12. 按「MVP Subagent 编排」调用 4 个 01 专用 subagent：前三个并行，第四个在拿到交互流程和 ASCII 草案后做质量审查。
13. 主会话合并 subagent 输出：以用户明确材料和已选答案为准，采用更具体、更克制、更能闭合 MVP 价值路径的建议；不得把 subagent 输出原文直接拼进 `prd.md`。
14. 若 `mvp-prd-quality-reviewer` 判断 `是否可直接写 PRD：否`，且原因是关键冲突或 ASCII 无法在当前信息下修正，则只追问一个最关键澄清问题；否则继续生成。
15. 阅读 `references/product-architecture-prompt.md`，基于 MVP Context Pack、subagent 输出和质量审查结论生成内部产品架构分析。
16. 内部架构分析只用于提升 PRD 质量，不作为独立章节输出给用户。
17. 完成内部架构分析后立即生成基础版 `prd.md`，不要询问、等待或要求用户上传图片。
18. 若在可写的本地文件系统中，将最终内容写入当前工作区的 `prd.md`；否则输出完整 Markdown 文本。
19. 基础版 `prd.md` 生成之后，在对话中提示用户可以上传一张不错的参考图来提炼设计系统；该提示不得写入 `prd.md`。
20. 基础版 `prd.md` 生成之后，只要用户上传图片，就阅读 `references/design-system-prompt.md`，提炼设计系统，并把可选的 `【UI 设计风格】` 章节追加到已有 `prd.md` 末尾。

六个维度的充分度标准、追问优先级和选项撰写规则见 `references/question-strategy.md`。

## 信息充分度状态

动态补洞不是看到信息就跳过，而是判断该信息是否已经具体到能决定 MVP 边界。每次追问前后，都在内部更新以下状态，不要把状态原文输出给用户：

- 已确认信号：用户初始想法、附件材料、图片信号、已选答案、自定义补充、题型选择。
- 工具形态边界：最终确认的入口、设备、平台能力、交互限制和默认使用场景。
- 未选答案信号：用户没有选择的选项代表的方向暂时降权；它们不是绝对否定，但不能在后续问题或 PRD 中继续作为默认主线。
- 剩余关键不确定性：下一题应该优先澄清当前最影响 MVP 边界的一个维度。
- 选项生成约束：题干和四个选项必须同时贴合「产品想法 + 已确认维度 + 未选答案降权 + 当前待补洞维度」。

使用未选答案时要克制：只把未选项当作弱信号，避免过度推断用户讨厌某类方向；但如果一个未选项对应明显不同的平台、场景或人群，就不要在后续追问里继续围绕它展开。

## MVP Subagent 编排

本 Skill 使用 4 个 01 专用 MVP subagent 做 PRD 生成前审查，不复用 `02-project-prepare` 的开发规格 subagent。`01` 的 subagent 只判断「这个 MVP 应该是什么」，不输出开发 TODO、技术架构或测试计划。

### 可调用 Subagents

| name | 职责 |
| --- | --- |
| `mvp-user-scenario-analyst` | 检查目标用户、使用场景、核心痛点是否具体且一致 |
| `mvp-scope-architect` | 收敛 MVP 功能边界，标注模块优先级和暂不做范围 |
| `mvp-interaction-flow-designer` | 设计真实入口、核心交互流程、界面/组件和 ASCII 草案 |
| `mvp-prd-quality-reviewer` | 审查整体一致性，并重点优化 ASCII 流程图的信息密度 |

### 调用顺序

1. 主会话生成 MVP Context Pack。
2. 并行 spawn：
   - `mvp-user-scenario-analyst`
   - `mvp-scope-architect`
   - `mvp-interaction-flow-designer`
3. 收集前三个 subagent 输出后，再 spawn `mvp-prd-quality-reviewer`，并把前三个输出摘要和 ASCII 草案一并传入。
4. 如果当前运行环境没有可用 subagent 工具，主会话必须按四个 agent 文档的职责顺序做同等自审；不得跳过 `mvp-prd-quality-reviewer` 的 ASCII 质量检查。

### Prompt 要求

每个 subagent 的 prompt 必须自包含，至少包含：

- 初始产品想法和用户材料摘要。
- 最终确认的工具形态、目标用户、使用场景、核心痛点、MVP 功能、交互流程。
- 已问澄清题、用户答案和未选答案降权信号。
- 图片信号摘要（如有）。
- 明确说明：只输出标准格式，不写最终 `prd.md`，不写开发规格，不 spawn 其他 subagent。

### 合并规则

- 用户明确表达和已选答案优先级最高。
- 采用更具体、更能闭合用户价值路径的建议。
- 功能范围以 `mvp-scope-architect` 的 4 分及以上模块为主；3 分及以下不得写成 MVP 主线。
- 交互入口以 `mvp-interaction-flow-designer` 为主，但必须接受 `mvp-prd-quality-reviewer` 对 ASCII 的压缩或增强建议。
- ASCII 流程图太简单时，补足页面/组件/操作/数据流；太复杂时，只保留核心路径、2-4 个主要分组和 4 分及以上模块。
- Subagent 输出是内部材料，不要作为独立章节写入最终 PRD。

## 空启动流程

当用户只输入技能名、只说「开始」「帮我生成 PRD」或没有给出明确产品想法时，不要直接生成工具形态题。

先阅读 `references/idea-inspiration.md`，并输出：

```text
你想做什么产品？

可以直接用一句话描述，例如：我想做一个猫咪爱好者的社交应用。

或者试试这些灵感：
- {{灵感1}}
- {{灵感2}}
- {{灵感3}}
- {{灵感4}}
- {{灵感5}}

请回复你的产品想法，我会先判断哪些信息已经足够，再只追问必要的问题来生成 prd.md。
```

等待用户给出产品想法后，再进入信息充分度评估。

## AskUserQuestion / AskQuestion 工具策略

出选择题时优先调用当前 Agent 环境对应的提问工具，而不是直接输出纯文本题。

工具命名规则：

- Claude Code / Claude Agent SDK 环境使用工具名：`AskUserQuestion`。
- Codex 环境使用工具名：`AskQuestion`。
- 工具白名单、allowed tools、canUseTool 或同类配置中，如果需要跨环境复用本技能，应同时允许 `AskUserQuestion` 和 `AskQuestion`；如果配置只服务单一环境，只允许该环境对应的工具名即可。
- 禁止使用 `askUserQuestion`、`askQuestion` 等小驼峰名称作为工具名；工具名大小写敏感。

工具调用要求：

- 每次只调用一次可用的提问工具，对应当前一道澄清题。
- 题目必须包含 `第 N / 最多 6 题` 和题型：`单选题` 或 `多选题`。
- 每题必须提供恰好 4 个选项。
- 单选题必须只允许选择一个选项。
- 多选题必须允许选择多个选项。
- 必须允许用户填写自定义答案。
- 选项值保留 A/B/C/D 标识，便于后续记录问答历史。

如果当前 Agent 环境没有暴露本环境对应的提问工具，或可用工具不支持多选/自定义输入，则回退为下面的文本式题目格式。

## 出题约定

当无法调用 `AskUserQuestion` 或 `AskQuestion` 时，每道题只输出题干与选项界面，格式如下：

```text
第 N / 最多 6 题（单选题|多选题）
问题文本

A. 选项一
B. 选项二
C. 选项三
D. 选项四

单选题：请回复一个选项字母，也可以直接写自定义答案。
多选题：请回复多个选项字母，例如 A、C，也可以直接写自定义答案。
```

不要在同一条消息里问多道题。只要仍有会影响 MVP 边界的关键缺口，不要提前展示 PRD 草稿；一旦六个维度都已充分，不要继续追问。

## 内部产品架构分析

信息足够后，必须先阅读 `references/product-architecture-prompt.md`。

分析目标：

- 从用户核心需求出发，而不是从技术实现出发。
- 先形成产品整体框架和核心业务逻辑。
- 区分界面层、组件层、业务流程层、数据层。
- 列出关键界面/页面、功能模块、可复用组件和核心数据对象。
- 用 MECE 原则筛选模块，避免功能重叠或遗漏。
- 给核心模块做优先级判断，只把优先级 4 分及以上的模块放进最终 ASCII 流程图。

映射规则：

- `工具形态:` 直接来自最终确认的工具形态，放在最终 `prd.md` 的 `Slogan:` 和 `【用户画像】` 之间。
- `【用户画像】` 来自目标用户、核心需求和主要痛点。
- `【场景故事】` 来自核心业务流程和高频价值路径。
- `【核心功能 MVP】` 来自优先级最高的界面层、组件层和业务流程层模块，默认收敛为四条。
- `【交互流程】` 来自从首页/主界面开始的用户核心旅程。
- `【ASCII 流程图】` 展示核心界面、关键组件、业务流程和数据流，不展示低优先级模块。

不要把内部分析中的「界面层」「组件层」「业务流程层」「数据层」作为新的 PRD 章节输出。

## 可选设计系统

设计系统生成不是额外澄清题，不要强制用户上传图片，也不要在生成基础版 PRD 前等待图片。

基础版 `prd.md` 生成并交付后，在对话里追加一句提示：

```text
如果你有一张觉得不错的产品或界面参考图，可以直接上传，我会提炼 UI 设计系统并追加到 prd.md 末尾。
```

这句话只作为对话提示，不得写入 `prd.md`。

触发条件：

- 基础版 `prd.md` 生成后，用户上传设计参考图。
- 用户明确要求「根据参考图生成设计系统」「分析 UI 风格」「把设计风格加入 PRD」。

执行规则：

- 先阅读 `references/design-system-prompt.md`。
- 基于用户上传的图片生成精简 UI 设计系统，控制在 200 字以内。
- 图片上传本身就是触发信号；无需再次询问用户是否分析。
- 只在成功生成设计系统时，把 `【UI 设计风格】` 追加到已有 `prd.md` 的 `【ASCII 流程图】` 之后。
- 如果用户没有上传图片，也没有要求设计系统，不要添加 `【UI 设计风格】`。
- 如果用户说跳过设计风格，直接保留基础版 `prd.md`。

## 最终 PRD 约定

生成终稿前先阅读 `references/prd-template.md`。

最终 `prd.md` 必须严格按以下章节顺序与标题文字：

```text
产品名称:
Slogan:
工具形态:

【用户画像】

【场景故事】

【核心功能 MVP】

【交互流程】

【ASCII 流程图】
```

如果存在图片生成的设计系统，唯一允许追加的可选章节是：

```text
【UI 设计风格】
```

硬性规则：

- 不要添加 `# 产品需求文档` 或任何 Markdown 总标题。
- 不要添加诸如 背景与目标、成功指标、风险、页面需求、数据需求、验收标准、待确认问题 等章节。
- 除可选的 `【UI 设计风格】` 外，不要添加其他新章节。
- 不要输出 JSON。
- 除非用户明确要求在行内查看文档，否则不要用代码围栏包裹整份终稿。
- 不要在 `prd.md` 内写说明、摘要或收尾话。
- 除产品名等必须用英文的专有名词外，全文使用中文。
- `【ASCII 流程图】` 下方必须使用 `text` 代码围栏包裹，开始行为 ```` ```text ````，结束行为 ```` ``` ````；不要输出 Mermaid 语法。
- ASCII 流程图使用文本分组表达核心界面/页面及其内部关键组件或功能模块。
- ASCII 流程图只包含内部架构分析里优先级评分为 4 分及以上的核心模块。
- ASCII 流程图必须经过 `mvp-prd-quality-reviewer` 的信息密度检查；太简单时补足入口、组件、操作、结果反馈和数据流，太复杂时压缩到核心路径。
- ASCII 流程图的结构符号只使用 ASCII 字符，例如 `->`、`|`、`+--`、`[页面]`；节点文字可以使用简洁中文。

## 生成指引

`工具形态:` 必须直接使用最终确认的工具形态，例如 `网站，优先在桌面使用`，放在 `Slogan:` 之后、`【用户画像】` 之前，不要改写成技术方案或平台建议。若用户只说「网站」「App」「平台」「工具」「系统」，不能视为工具形态已充分。

根据用户想法与确认信息，推断具体的产品名称与 Slogan。保持 MVP 务实、克制。

用户画像要具体：含姓名、年龄、身份或角色、日常处境、痛点与动机。场景故事要具象，尽量包含时间/地点/触发事件、动作和结果。

`【核心功能 MVP】` 默认输出四条要点，除非用户材料和追问答案明显支持更少。每条必须使用 `- 功能名：功能说明`。

`【交互流程】` 用简洁的要点步骤，且能直接映射到 ASCII 流程图。ASCII 流程图要突出从首页/主界面开始的用户核心旅程，并标注关键用户操作路径、数据流向和复用组件出现位置。
