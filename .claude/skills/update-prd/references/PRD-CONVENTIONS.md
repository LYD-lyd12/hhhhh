# PRD 写作规范

未来 PRD 分为两层：产品层和技术层。产品层让人快速理解模块状态，技术层让开发人员、维护者和后续 Agent 能继续开发、调试和验证。

## 推荐目录

```text
prd/
  README.md
  .update-prd-state.json
  <module>/
    README.md
    technical.md
    sync.json
    decisions.md
```

说明：

- `prd/README.md`：项目总览和模块索引，给人快速了解全局。
- `prd/<module>/README.md`：模块产品文档，描述产品状态和用户流程。
- `prd/<module>/technical.md`：模块技术文档，描述实现上下文和调试信息。
- `prd/<module>/sync.json`：模块级同步状态。
- `prd/<module>/decisions.md`：可选，用于记录长期有效的关键产品/技术决策。

## 产品 README

目标：让产品、设计、开发新人或项目 owner 在 2 分钟内看懂。

建议章节：

1. **模块定位**：这个模块解决什么问题，为什么存在。
2. **当前已实现**：用户现在能做什么，入口在哪里，主流程是什么。
3. **还没做什么**：只列关键缺口，详细任务放 `TODO.md`。
4. **关键决策**：为什么这样做，哪些方案暂时没选。
5. **关联模块**：与其他模块的产品关系。

写作要求：

- 少写代码细节，不贴大段接口和类型。
- 用产品语言解释能力和状态。
- 每个模块 README 默认控制在可快速浏览的长度；内容过多时拆到 `technical.md` 或 decisions。
- 不把未验证能力写成已实现。
- 不把原始 `prd.md` 目标直接搬成当前事实。

## 技术文档

目标：让开发人员、维护者和后续 Agent 能准确定位代码、继续开发和调试。

建议章节：

1. **代码入口**：页面、组件、API、IPC、后台任务、配置文件。
2. **数据结构**：核心类型、schema、状态字段、持久化位置。
3. **调用链路**：API/IPC/第三方服务请求、关键参数摘要、成功/失败结果。
4. **日志要求**：调用入口、关键参数摘要、成功/失败、错误原因；不得记录 API Key、Token、用户隐私原文。
5. **测试与验收**：测试文件、`data-testid`、手动验收步骤、已知跳过原因。
6. **边界情况**：降级逻辑、空态、错误态、权限、网络失败、浏览器/系统差异。
7. **TODO 对应项**：与 `TODO.md` 的任务编号或条目关系。
8. **最近同步摘要**：本轮 diff 影响、同步 commit、未确认点。

## 代码引用

- 使用相对仓库根目录的路径：`app/page.tsx`、`src/lib/store.ts`。
- 产品 README 只列关键入口和少量核心文件。
- 技术文档可列完整相关文件，并说明每个文件为什么重要。
- 写入前快速确认路径存在；不存在时标注“待确认”。

## 流程图

- 产品 README 可用简短 ASCII 流程图说明用户流程。
- 技术文档可用更详细流程图说明调用链。
- 不强制使用 Mermaid；若项目当前规范要求 ASCII，遵守项目规范。

## TODO 差距

`TODO.md` 是未完成任务入口。以下内容必须写回 TODO：

- 原始目标中未实现或部分实现的功能。
- 代码中发现但 PRD 未覆盖的后续任务。
- 验证失败、跳过或阻塞项。
- 用户明确延后的功能。

PRD 可以简要写“还没做什么”，但具体下一步要落在 TODO 中。
